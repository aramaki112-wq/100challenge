# Thought Process.md
# DAY17 Release Phase1
## 状態を持つから、状態を設計に使うへ

---

> [!NOTE]
> ## 今日のテーマ
>
> 今日の目的は機能を追加することではない。

# 今日最初に考えたこと

昨日までのタスク管理アプリは問題なく動いていた。

タスクを追加できる。

削除できる。

編集もできる。

それだけ見れば完成しているように思える。

でも、少し未来を想像してみた。

検索機能を追加したい。

並び替えも欲しい。

カテゴリも付けたい。

お気に入り機能も面白そうだ。

その瞬間、ある疑問が浮かんだ。

「その状態は、どこに保存するのだろう？」

---

# フィルターはタスクなのか？

最初に頭に浮かんだのは、

tasks の中へ currentFilter を入れる方法だった。

でも、すぐに違和感があった。

例えば、

「牛乳を買う」

というタスクがある。

そこへ

「完了だけ表示中」

という情報を入れる。

……何かがおかしい。

それはタスク自身の情報ではない。

画面の状態だ。

つまり、

タスク(Data)

と

画面(State)

は別物なのである。

---

> [!TIP]
> ## 今日の気付き
>
> Dataは「何を持っているか」
>
> Stateは「どう見せるか」

---

# Stateという箱を作る

変数を一つ作れば十分かもしれない。

```javascript
let currentFilter = "all";
```

でも、このコードを見ながら考えた。

今日必要なのは filter だけ。

では、明日は？

検索。

並び替え。

カテゴリ。

編集中かどうか。

テーマカラー。

状態は必ず増える。

だったら最初から

```javascript
const state = {};
```

という箱を用意しておこう。

今は一つしか入っていなくても、

未来ではその箱が活躍する。

---

# init() を最初に作る理由

コードを書き始める前に、

入口を決めたかった。

アプリは必ず

「起動」

から始まる。

その入口が散らばると、

あとで自分でも読めなくなる。

だから

```javascript
init();
```

だけは最初に決める。

そして、その中には

「何をするか」

だけを書く。

「どうやるか」

は別の関数へ任せる。

これで入口は一つになる。

---

# イベントは仕事をしない

イベントを書くと、

つい処理を書きたくなる。

昔の自分もそうだった。

クリックされた。

↓

その場で全部処理する。

でも実務では違う。

イベントは

「担当者を呼ぶ」

だけ。

つまり、

```text
イベント

↓

changeFilter()
```

まで。

仕事そのものは

changeFilter() が担当する。

役割を分ける。

それだけでコードは驚くほど読みやすくなる。

---

# あえて表示を書かない

今日は displayTasks() を書ける。

render() も書ける。

でも書かない。

理由は簡単。

設計を先に完成させたいから。

表示を書き始めると、

画面は動く。

しかし、

設計は見えなくなる。

今日は

State が変わる。

そこまで。

表示は次のPhaseへ譲る。

---

> [!IMPORTANT]
> ## 今日の設計判断
>
> 「今動くコード」より
>
> 「あとで育てられるコード」を優先する。

---

# 今日一番大事だったこと

State を作ったことではない。

Object を作ったことでもない。

本当に大事だったのは、

役割を分けたことだった。

Data

State

Event

Initialize

DOM

それぞれが

自分の仕事だけを持つ。

これが責務分離である。

---

# 明日への橋渡し

今日作ったコードは、

まだ静かな土台でしかない。

State は変わる。

でも画面は変わらない。

Data は読み込める。

でも追加できない。

次はいよいよ、

この土台の上へ

CRUDを積み上げる。

追加。

更新。

削除。

完了。

ここからアプリは

「設計された土台」

の上で成長を始める。

---

# 今日の独り言

プログラムを書くことは、

家を建てることによく似ている。

壁を作る方が楽しい。

屋根を付ける方が達成感もある。

でも、

基礎工事を省いた家は、

大きくなった瞬間に崩れてしまう。

DAY17は、

誰も気付かない基礎工事の日だった。

派手さはない。

画面もほとんど変わらない。

それでも、

今日作った設計は、

これから先の何十日もの開発を支える土台になる。

未来の自分が、

「この設計で良かった」

そう思えるように、

今日は機能ではなく、

設計を選んだ。
